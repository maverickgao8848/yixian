# /claudemd-updater — 从失败中提炼工程原则

把本次 session 的教训变成未来不再犯的规则。

## 触发场景

- Claude 卡在某个错误循环中出不来
- Claude 的产出和用户预期有明显偏差
- 用户发现 Claude 犯了本不该犯的错
- 用户希望把一条经验"写进系统"防止再犯
- 用户希望同步 CLAUDE.md 的目录结构描述与实际仓库结构（触发词：同步目录、更新目录结构、sync structure）

## 工作流程

### Phase 1: 收集反馈

先问用户一个简洁的问题（如果用户还没说清楚）：

> 出了什么问题？你期望的结果 vs 实际结果是什么？

用一句话引导用户描述偏差，不需要长篇大论。

### Phase 2: 回溯诊断

**浏览本次 session 的对话记录**，从以下角度分析：

1. **根因定位**：Claude 在哪一步开始偏离？是理解偏差、工具使用错误、还是缺少某个约束？
2. **缺失工具/原则识别**：如果 CLAUDE.md 里有一条规则，是否能阻止这次错误？
3. **模式归类**：这是偶发还是系统性问题？属于哪类：
   - **理解偏差** — Claude 误解了意图
   - **执行偏差** — 理解对了但做错了（工具、路径、语法等）
   - **边界缺失** — 没有约束导致 Claude 自由发挥过头
   - **重复犯错** — 同类错误出现 ≥2 次

### Phase 3: 提炼原则

从诊断结果中提炼 **一条精要原则**，要求：

- **可操作**：能直接指导未来行为，不是空话（"要小心" ✗ → "修改 X 类文件前必须先 Read" ✓）
- **有边界**：明确适用范围，防止过度约束（"只适用于 Y 场景"）
- **精简**：一句话核心 + 可选的一句补充，总长不超过 3 行

原则格式：
```
### {简短标题}
{一句话核心规则}。**场景**: {适用范围}。
```

### Phase 4: 确认并写入

1. 向用户展示提炼的原则草案
2. 等用户确认或修改
3. 用户确认后，执行写入：
   - 读取当前 `CLAUDE.md`
   - 找到合适的插入位置（按相关性归入已有 section，或新建 section）
   - 插入原则，保持文档整体结构
   - 展示写入后的 diff 摘要

## 写入规则

- **不删除**已有原则，只追加
- **不重复**：写入前检查 CLAUDE.md 中是否已有相同或高度相似的原则；如有，提出合并建议而非重复添加
- **位置就近**：新原则放入语义最接近的 section；没有合适的 section 时，在 `开发循环守则` 之后新增 section
- **格式一致**：使用与 CLAUDE.md 现有内容一致的 Markdown 风格（中文、简洁列表）

## 输出格式

```markdown
## 诊断结果

**问题类型**: {理解偏差 | 执行偏差 | 边界缺失 | 重复犯错}
**根因**: {一句话}

## 建议新增原则

### {标题}
{一句话核心规则}。**场景**: {适用范围}。

---
确认写入 CLAUDE.md？如有修改意见请直接说。
```

## 目录结构同步模式

扫描当前项目目录，将 CLAUDE.md 中的「仓库目录结构」section 与实际文件系统同步。

### 触发条件

用户输入包含以下任一关键词时激活此模式（而非原则提炼模式）：
- `同步目录`、`更新目录结构`、`sync structure`

### 步骤 1: 扫描目录

1. 使用 `find` 或 Glob 工具扫描项目根目录（即 CLAUDE.md 所在目录）
2. 深度 3-4 层，排除 `node_modules`、`.git`、`__pycache__`、`.next`、`dist`、`build` 等生成物目录
3. 生成标准 tree 格式输出，每个目录/文件附带简短中文注释（与现有 CLAUDE.md 风格一致）

### 步骤 2: 校验基本骨架

检查以下必要路径是否存在：

| 必须存在 | 说明 |
|---|---|
| `CLAUDE.md` | 开发协议 |
| `.agents/skills/` | skill 定义目录 |
| `.agents/workflows/` | 工作流定义目录 |
| `docs/` | 项目文档 |
| `projects/` | 项目目录 |
| `tools/` | 工具目录 |

- **全部存在** → 进入步骤 3
- **有缺失** → **立即终止**，向用户报告缺失目录列表，建议先创建对应目录或调整结构

### 步骤 3: 生成更新后的目录树

基于扫描结果，生成新的 `## 仓库目录结构` section 内容，遵循以下规则：

- 使用 ``` 包裹的 tree 格式
- 每行带 `# 中文说明` 注释
- 目录用 `/` 结尾
- 细化到有意义的子目录和关键文件（如 `SKILL.md`、配置文件等）
- 过时的条目（CLAUDE.md 中写了但实际不存在的）移除
- 新增的条目补充进去

### 步骤 4: 确认并替换

1. 向用户展示变更摘要和 diff
2. 等用户确认
3. 用户确认后，使用 **Edit 工具**精确替换 CLAUDE.md 中 `## 仓库目录结构` 到下一个 `---` 之间的内容（包含紧随其后的 `---` 分隔线之后、下一个 `##` 之前的说明文字也一并保留）
4. 展示替换后的 diff 摘要

### 禁止规则（硬约束 — 任何情况下不得违反）

- **禁止修改** `三 Agent 开发架构` 及其所有子 section
- **禁止修改** `依赖层级` section
- **禁止修改** `开发循环守则` section
- **禁止添加、删除或修改** 目录结构 section 以外的任何内容
- **禁止**在基本骨架校验失败时继续执行

### 输出格式

```markdown
## 目录结构同步结果

**变更摘要**: 新增 {X} 个条目，移除 {Y} 个过时条目，更新 {Z} 个注释
**校验状态**: 基本骨架完整 ✓

```diff
- {被移除的旧条目}
+ {新增的条目}
  {未变化的条目}
```

确认写入 CLAUDE.md？如有修改意见请直接说。
```
