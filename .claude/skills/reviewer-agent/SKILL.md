# /reviewer-agent — 多维度代码审查系统

像资深 Tech Lead 一样帮你审查代码 🔍

## 触发词

- "代码审查"
- "review"
- "检查代码"
- "PR审查"
- "代码质量"
- "帮我看看这段代码"

## 使用场景

- Pull Request 审查
- 代码质量检查
- 上线前安全审查
- 重构前评估

## 工作流程

### Phase 1: 审查范围确认 🎯

**输出格式**：

```markdown
## 🔍 审查计划

**审查目标**：{file_path 或 PR}

**审查范围**：
- 文件数：{n}
- 代码行数：{n}
- 语言/框架：{...}

**启用的审查维度**：

| 维度 | 重点 | 严格程度 |
|------|------|---------|
| 🔒 安全性 | {...} | {标准} |
| ⚡ 性能 | {...} | {标准} |
| 📋 代码质量 | {...} | {标准} |
| 🔧 可维护性 | {...} | {标准} |

---
📍 开始执行审查...
```

---

### Phase 2: 并行专业审查 🚀

#### 2.1 Security Agent（安全审查）

**检查项**：注入漏洞、认证授权、数据安全、依赖安全、配置安全

**输出格式**：

```markdown
## 🔒 安全审查报告

**风险等级**：🔴 高 / 🟡 中 / 🟢 低

### 🔴 严重问题

1. **{问题标题}**
   - 位置：`{file}:{line}`
   - 风险：{具体风险描述}
   - 建议：{修复方案}
```

#### 2.2 Performance Agent（性能审查）

**检查项**：算法复杂度、内存使用、I/O 优化、渲染性能、缓存策略

#### 2.3 Quality Agent（代码质量审查）

**检查项**：代码规范、错误处理、测试覆盖、类型安全、代码重复

#### 2.4 Maintainability Agent（可维护性审查）

**检查项**：架构设计、文档完整、可扩展性、配置管理、日志监控

---

### Phase 3: 综合评估与优先级排序 📊

**优先级定义**：

| 级别 | 标签 | 含义 | 行动 |
|------|------|------|------|
| P0 | 🔴 阻塞 | 严重问题 | 阻止合并 |
| P1 | 🟠 重要 | 重要问题 | 强烈建议修复 |
| P2 | 🟡 建议 | 改进建议 | 创建跟进 Issue |
| P3 | 🟢 可选 | 优化建议 | 可忽略 |

**输出格式**：

```markdown
## 📊 综合评估

### 整体评分

| 维度 | 评分 | 状态 |
|------|------|------|
| 🔒 安全性 | {n}/10 | {状态} |
| ⚡ 性能 | {n}/10 | {状态} |
| 📋 代码质量 | {n}/10 | {状态} |
| 🔧 可维护性 | {n}/10 | {状态} |
| **综合** | **{n}/10** | **{总体状态}** |

### 审查结论

{🟢 可以合并 / 🟡 建议修复后合并 / 🔴 需要重大修改}
```

---

### Phase 4: 审查报告输出 📝

**输出位置**：`output/reviews/{target}/report.md`

**报告结构**：
1. 执行摘要
2. 综合评估
3. 各维度详细审查
4. 行动清单（按优先级）
5. 亮点

## 审查维度自动选择

| 代码类型 | 安全 | 性能 | 质量 | 可维护性 |
|---------|------|------|------|---------|
| 前端 (JS/TS/React) | ✓ | ✓ | ✓ | ✓ |
| 后端 (API/服务) | ✓✓ | ✓ | ✓ | ✓ |
| 数据库 (SQL) | ✓ | ✓✓ | ✓ | ✓ |
| 配置文件 | ✓ | - | ✓ | ✓ |
| 测试文件 | - | - | ✓✓ | ✓ |

## 最佳实践

### 1. 建设性反馈

```
❌ "这段代码写得不好"

✅ "建议将此函数拆分，当前函数有 80 行，承担了多个职责。
拆分后可提高可测试性和可读性。"
```

### 2. 提供代码示例

每个问题包含：问题代码 + 建议修改 + 原因说明

### 3. 识别亮点

不只找问题，也要肯定做得好的地方
